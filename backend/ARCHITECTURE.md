# Архитектура бэкенда «Отчёты АТЦ»

Источник истины по бизнес-логике — `../ТЗ_Отчёты_АТЦ.md`. Этот файл описывает техническую реализацию.

## Роли в разработке
- **Архитектор (Claude):** проектирование, спецификации, финальное тестирование и шлифовка.
- **DeepSeek (агент):** генерация кода и сборка по спецификации.

## Стек и принципы
- Python 3.11+, виртуальное окружение `venv`, зависимости в `requirements.txt`.
- HTTP-клиент: `requests`. Excel: `openpyxl`. Конфиг: `python-dotenv`.
- Вся логика — в бэкенде (см. решение по независимости от 1С: только чтение OData).
- Секреты — в `.env` (в git не коммитятся). В репозитории — `.env.example`.
- Никаких записей в 1С. Только GET-запросы. Пользователь 1С — READ ONLY.

## Схема данных
```
1С OData (Basic auth, логин UTF-8)
  → ODataClient (чтение)
  → Repositories (заказ-наряды, регистр «Доходы и расходы», план, выписки, счета-фактуры)
  → MetricsService (расчёт показателей ПЛАН/ФАКТ по разделам 4–5 ТЗ)
  → ExcelReporter (заполнение шаблона)
  → BitrixSender (отправка)  ← позже
  → Scheduler (cron/APScheduler: ежедневно 11:00, пятница 11:00) ← позже
```

## Параметры подключения к 1С (тестовая база)
- Base URL: `http://91.144.178.239:8888/test_moskva/odata/standard.odata/`
- Auth: Basic, **логин в кодировке UTF-8**
- Кириллица в путях — percent-encode (`urllib.parse.quote`)
- Формат: `?$format=json`
- Доступ к порту 8888 нестабилен (IP-allowlist) — клиент должен делать ретраи с таймаутом и понятной ошибкой.

## План инкрементов
1. **Фундамент (сейчас):** scaffold, config, OData-клиент, references, smoke-test чтения реальных данных.
2. Repositories + MetricsService (дневные показатели Арсенал/Реф) на реальных данных.
3. ExcelReporter — заполнение шаблона (5×2), проверка формул.
4. Остальные направления (Шаркер/ЦКР/Мойка), недельные формы.
5. BitrixSender + Scheduler + логирование.

## Инкремент 1 — критерии приёмки
- `python -m venv` + `pip install -r requirements.txt` проходят.
- `python src/smoke_test.py` при живом OData:
  1. подключается, печатает число EntitySet;
  2. читает `Catalog_ПодразделенияКомпании` и подтверждает наличие всех 6 названий из маппинга (сообщает, какие не найдены);
  3. читает `Catalog_ВидыСостоянийЗаказНарядов` и печатает список статусов;
  4. читает `Document_ЗаказНаряд` с фильтром по дате (последние 30 дней) и печатает количество + 1 пример (номер, дата, подразделение, статус, сумма);
  5. при недоступности порта — понятное сообщение и ненулевой код возврата.
- Секреты только из `.env`; в коде хардкода паролей нет.
