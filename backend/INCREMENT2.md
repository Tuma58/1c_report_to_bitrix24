# Инкремент 2 — Repositories + расчёт ДНЕВНЫХ показателей (Арсенал, Реф.Сервис)

Основано на разведке живой базы `test_moskva` (архитектор). Скоуп: только дневные показатели для «Арсенал» и «Реф. Сервис». Остальные направления/недельные — следующие инкременты.

## Модель данных (проверено на реальных данных)

### Заказ-наряды
- `Document_ЗаказНаряд`: `Ref_Key`, `Date`, `Number`, `ПодразделениеКомпании_Key`, `Состояние_Key`, `ДатаЗакрытия`, `СуммаДокумента`.
- Закрытый ЗН за день D: `Состояние_Key eq guid'<Закрыт>' and ПодразделениеКомпании_Key eq guid'<div>' and ДатаЗакрытия ge datetime'D' and ДатаЗакрытия lt datetime'D+1'`.
- Ключ статуса «Закрыт» берём из `References.statuses()['Закрыт']`.

### Нормочасы (табличная часть)
- `Document_ЗаказНаряд_Работы`: строки, ключ `Ref_Key` = ссылка на ЗН. Поля `Количество`, `Коэффициент`.
- **Нормочасы строки = Количество × Коэффициент.** Выработка ЗН = сумма по строкам.
- Читать: для набора закрытых ЗН за день фильтровать `Работы` по их `Ref_Key` (группами через OR, батч ~20 guid на запрос).

### Исполнители
- `Document_ЗаказНаряд_Исполнители`: поля `Ref_Key`, `Исполнитель_Key`.
- Кол-во мастеров = число уникальных `Исполнитель_Key` по набору закрытых ЗН за день.

### Выручка / себестоимость — регистр «Доходы и расходы»
- EntitySet: **`AccumulationRegister_ДоходыИРасходы_RecordType`** (НЕ базовый `..._ДоходыИРасходы` — он даёт только Recorder/RecordSet).
- Поля: `Period`, `ПодразделениеКомпании_Key`, `СтатьяДоходовИРасходов_Key`, `ДоходБезНДС`, `РасходБезНДС`, `Active`.
- Фильтр по дню: `ПодразделениеКомпании_Key eq guid'<div>' and Period ge datetime'D' and Period lt datetime'D+1'`.
- **Выручка за день (без НДС)** = Σ `ДоходБезНДС` по статьям `{«Выручка по заказ-нарядам», «Работы по заказ-нарядам»}`.
- **Себестоимость ЗЧ (без НДС)** = Σ `РасходБезНДС` по статье `«Себестоимость материалов»`.
- Статьи резолвить через `Catalog_СтатьиДоходовИРасходов` (Description→Ref_Key), добавить в `references.py` метод `income_expense_articles()`.
- ВАЖНО: брать только перечисленные статьи; НЕ включать «Корректировка баланса по подразделениям» и прочие.

### План (месячный документ)
- `Document_ПрибылиУбытки_УстановкаЗначенийПлановыхПоказателей`: `Ref_Key`, `ПодразделениеКомпании_Key`, `ДатаФиксации` (месяц), `Date`.
- Табличная часть `..._Показатели`: `Ref_Key`(=ЗН документа), `Показатель`(guid→`Catalog_ПрибылиУбытки_ПлановыеПоказатели`), `ЗначениеПоказателя`.
- Выбор документа: последний по `Date` для подразделения, у которого `ДатаФиксации` в месяце отчётного дня.
- **Пересчёт в день:** значение месяца ÷ число дней в месяце (раздел 3.3 ТЗ).

## Маппинг план-показателей → строки отчёта (ПРОВЕРИТЬ у владельца/1С)
В `Catalog_ПрибылиУбытки_ПлановыеПоказатели` под проект заведены (коды 67–74). Предлагаемый маппинг для дневного Арсенал/Реф — вынести в КОНФИГ (dict «строка отчёта → Code показателя»), чтобы менять без правки кода:

| Строка отчёта (ПЛАН) | Предлагаемый показатель | Code |
|---|---|---|
| Выручка за день | Выручка за месяц | 000000067 |
| Себестоимость ЗЧ | Себестоимость запасных частей | 000000070 |
| Выработка на 1 мастера | Выработка на 1 мастера план в месяц | 000000068 |
| Средний чек (для «Закрыто ЗН план») | Средний чек | 000000069 |
| Нормочасы за день — общая выработка | Нормочасы факт/день (?) | 000000071 |

- «Закрыто ЗН за день ПЛАН» = Выручка_план_день ÷ Средний_чек_план (раздел 4.1 ТЗ).
- Наценка ПЛАН = Выручка_план − Себестоимость_план.
- ⚠️ Маппинг помечен как черновой — построить его через конфиг, реальные Code подставит архитектор после подтверждения. Пока не подтверждено — ПЛАН может возвращать None и в отчёте прочерк.

## Что сгенерировать
```
src/repositories.py   # OrderRepository, IncomeExpenseRepository, PlanRepository
src/metrics.py        # DailyMetrics: dataclass результата + MetricsService.daily(division_report_name, date)
src/plan_mapping.py   # конфиг: строка отчёта -> Code план-показателя (по таблице выше)
tests/test_metrics_smoke.py  # прогон daily() для «Арсенал» и «Реф. Сервис» за конкретную дату
```

### MetricsService.daily(report_name, date) должен вернуть (ФАКТ + ПЛАН, где есть):
- нормочасы (ФАКТ = Σ Количество×Коэффициент; ПЛАН из конфига)
- выработка на 1 мастера (ФАКТ = нормочасы / кол-во исполнителей; ПЛАН)
- закрыто ЗН (ФАКТ = кол-во; ПЛАН = выручка_план/средний_чек)
- выручка (ФАКТ из регистра; ПЛАН)
- себестоимость ЗЧ (ФАКТ из регистра; ПЛАН)
- наценка (ФАКТ = выручка−себест; ПЛАН = аналогично)

## Критерии приёмки Инкремента 2
- `MetricsService.daily('Арсенал', date)` и `daily('Реф. Сервис', date)` отрабатывают на живой базе без ошибок.
- ФАКТ-значения ненулевые для дня с активностью (например, будний день июня 2026); печатается разбор (нормочасы, выручка, себестоимость, кол-во ЗН, кол-во мастеров).
- Выручка ФАКТ = сумма только двух статей регистра; себестоимость = статья «Себестоимость материалов».
- Нет записи в 1С; только GET.
- ПЛАН: если Code в конфиге не заполнен/не найден — возвращать None, отчёт не падает.
